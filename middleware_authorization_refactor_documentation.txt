=========================================================
Middleware Authorization Refactor Documentation
=========================================================

1. Files Modified
-----------------
- `routes/web.php`
- `bootstrap/app.php`
- `app/Http/Controllers/UserController.php`
- `app/Http/Controllers/RoleController.php`
- `app/Http/Controllers/ExpenseController.php`
- `app/Http/Controllers/ExpenseCategoryController.php`

2. Files Created
----------------
- `app/Http/Middleware/CheckPermission.php`

3. Middleware Explanation
-------------------------
The custom `CheckPermission` middleware was created to replace inline controller gate checks. It intercepts HTTP requests before they reach the controller, validates if the authenticated user has a specific permission based on their assigned role, and dynamically checks for resource ownership (bypasses) where necessary.

4. Permission Flow Explanation
------------------------------
When a user attempts to access a protected route:
1. The route triggers the `permission:<ability>` middleware.
2. The middleware verifies `Auth::check()`.
3. It checks if there's a dynamic ownership exception (e.g., a user editing themselves).
4. If no ownership exception applies, it invokes `$user->hasPermission('<ability>')`.
5. If the role possesses the permission, the request continues (`$next($request)`). If not, it aborts immediately with a 403 Forbidden.

5. Old vs New Authorization Approach
------------------------------------
- **Old Approach:** Requests passed freely to the Controller. Inside the controller methods, `Gate::authorize(...)` and manual `$user->id !== $authUser->id` checks were evaluated to grant or deny access.
- **New Approach:** Controllers are oblivious to authorization. The HTTP routing layer (Middleware) explicitly declares required permissions. The request is rejected at the perimeter before any controller logic executes.

6. Gate::authorize Replacement Strategy
---------------------------------------
All `Gate::authorize()` calls were completely removed from controllers. Their logic was replicated and abstracted into the `CheckPermission` middleware, utilizing Laravel's route definitions to explicitly map endpoints to permissions. Blade UI `@can` directives remain unaffected because `AppServiceProvider` still utilizes `Gate::before`.

7. Route Middleware Integration
-------------------------------
Routes now specify their required permission via the `->middleware('permission:<ability>')` method.
For routes requiring ownership checks (edit, update, destroy), a second parameter is passed to the middleware: `->middleware('permission:edit-expense,expense')`.

8. Permission Validation Logic
------------------------------
Inside the middleware, `Auth::user()->hasPermission($permission)` is called. This maps directly to the `User` model, traversing the `roleInfo` relationship, and checking if the `Role` model's `permissions` collection contains the required ability.

9. Security Considerations
--------------------------
- **Fail-Safe Denials:** The middleware strictly defaults to denying access (`abort(403)`) unless explicitly authorized.
- **Perimeter Defense:** Denying access at the middleware level prevents potential unauthorized data exposure, unexpected processing, or heavy database queries that could occur if unauthorized requests were allowed to reach the controller layer.
- **Soft-deletes Coverage:** Explicit numeric/string ID resolution is built-in to prevent bypassing logic when querying soft-deleted items during a "restore" action (which breaks standard implicit route model binding).

10. Step-by-Step Implementation Explanation
-------------------------------------------
Step 1: Analyzed all occurrences of `Gate::authorize` and how ownership overrides were implemented.
Step 2: Created `CheckPermission.php` with robust context-aware checks.
Step 3: Registered the `'permission'` alias in `bootstrap/app.php`.
Step 4: Attached `->middleware(...)` to all relevant protected routes in `web.php`.
Step 5: Purged all manual authorization logic and `Gate` facades from the controllers.

11. Code Explanations
---------------------
- `$request->route($modelKey)`: Fetches the bound model instance (e.g. `User` or `Expense`) directly from the incoming route parameter to evaluate ownership.
- `$next($request)`: A closure that allows the request to proceed deeper into the application (to the controller).
- `abort(403)`: Instantly halts execution, preventing further logic, and serves a standard HTTP 403 Forbidden response.

12. Laravel Middleware Lifecycle Explanation
--------------------------------------------
1. A request enters the application.
2. Global middleware executes (e.g., CSRF protection, session startup).
3. Route-specific middleware executes. Here, the `auth` middleware runs first to ensure the user is logged in.
4. Next, the `permission` middleware runs. If `CheckPermission` halts the request, the lifecycle ends here.
5. If `permission` calls `$next($request)`, the request finally reaches the Controller.
6. The Controller executes business logic and returns a Response, which passes back out through the middleware stack before being sent to the client's browser.
