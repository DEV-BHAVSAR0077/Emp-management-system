=========================================================
          LARAVEL EXPENSE MODULE DOCUMENTATION
=========================================================

This document provides a comprehensive, beginner-friendly explanation of the Expense Module built into this Laravel application. It outlines the architecture, step-by-step implementation, and the exact files created and modified.

---------------------------------------------------------
1. ARCHITECTURE OVERVIEW
---------------------------------------------------------
The Expense Module is designed to be fully isolated and modular. It adheres to Laravel's MVC (Model-View-Controller) architecture. It uses "soft deletes" (meaning deleted items are hidden rather than permanently removed from the database) and integrates seamlessly into the existing dashboard and navigation bar without breaking the current UI or requiring new libraries.

4. DATABASE MIGRATIONS EXPLAINED
---------------------------------------------------------
We created three tables to store our data properly:
1. `expense_categories`: The main buckets (e.g., "Travel", "Office Supplies").
2. `expense_sub_categories`: More specific buckets that belong to a parent category (e.g., "Flights" belongs to "Travel").
3. `expenses`: The actual logged expense, which tracks the amount, date, name, the user who created it, and links it to a category.

All tables use `$table->softDeletes()`. This means when a user clicks "Delete", Laravel just timestamps a `deleted_at` column instead of erasing the row permanently.

---------------------------------------------------------
5. MODEL RELATIONSHIPS
---------------------------------------------------------
In Laravel, Models talk to the database. We set up rules so they know how they relate to one another:
- Expense `belongsTo` a User, an ExpenseCategory, and an ExpenseSubCategory.
- We added `->withTrashed()` to these relationships. This is important! If a user deletes the "Travel" category, old expenses under "Travel" will still know they belong to "Travel" instead of breaking or showing blank spaces.
- ExpenseCategory `hasMany` ExpenseSubCategories.

---------------------------------------------------------
6. CONTROLLERS & ROUTES EXPLAINED
---------------------------------------------------------
Routes (`web.php`) map URLs to Controller actions.
- Web Routes (`ExpenseController`): When a user visits `/expenses/create`, Laravel calls the `create` method, which gathers categories and returns the HTML view. When the form submits, the `store` method validates the data and saves it.
- API Routes (`ExpenseCategoryController`): We added special routes like `/expense-categories/{id}/sub-categories` that return JSON instead of HTML. These allow our Javascript to quietly ask the server for data without refreshing the whole webpage.

---------------------------------------------------------
7. FRONTEND & DYNAMIC JAVASCRIPT LOGIC
---------------------------------------------------------
The frontend uses standard Blade templates (Laravel's HTML engine).
To make the Category and Sub-Category selection feel modern and dynamic without refreshing the page, we created `category_section.blade.php`. 

How it works step-by-step:
1. When a user selects a Category, plain Javascript catches the 'change' event.
2. It sends a hidden background request (fetch/AJAX) to our `ExpenseCategoryController`.
3. The controller returns the relevant Sub-Categories in JSON format.
4. The Javascript empties the old Sub-Category dropdown and fills it with the new options.
5. Users can also click a "+" button to type a new category. The Javascript sends this directly to the database and selects it instantly without losing the user's progress on the form.
6. To prevent bad user experiences, the Javascript temporarily disables the "Save" buttons and changes them to say "Saving..." while waiting for the server.

---------------------------------------------------------
8. SOFT DELETE & RESTORE SYSTEM
---------------------------------------------------------
Because data preservation is critical:
- When an expense is deleted, it just gets hidden (`deleted_at` is populated).
- On the dashboard's Expense tab, we added a small dropdown: "Active Expenses" vs "Deleted Expenses".
- If the user selects "Deleted Expenses", the URL changes, and `UserController` uses `->onlyTrashed()` to fetch only the hidden items.
- In this view, the "Edit/Delete" buttons are replaced by a "Restore" button. Clicking this triggers the `restore()` method in the controller, making the expense visible again.

---------------------------------------------------------
9. NAVBAR INTEGRATION
---------------------------------------------------------
To seamlessly add the module to the app:
- We located `navbar.blade.php` and copied the design of the "Roles" tab.
- We added an `@can('view-expense')` rule, meaning only users with the HR/Admin permission to view expenses will even see the button.
- We modified `Navbar.php` (a Laravel View Component) to track when the user is on any `/expenses` URL, so it can apply the visual blue underline ("active state") to the navbar tab automatically.
